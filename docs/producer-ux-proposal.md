# Producer UX Proposal: Zero-Config Default Path

**Status**: PROPOSAL (not yet implemented)
**Date**: 2026-03-21
**Author**: Architecture review of existing CLI + node codebase

---

## Problem Statement

A producer joining the DOLI network today must understand:
- Which commands need `sudo` and which do not
- That `~` resolves differently when the service runs as root vs. a user
- That `--producer-key` requires an absolute path in the service file
- That wallet, data, and key files live in different places depending on who installed what

This proposal defines a single, deterministic directory layout and permission model so that every default path "just works" -- for both the CLI (run interactively) and the node (run as a systemd/launchd service).

---

## Design Principles

1. **Zero flags for the common case.** A single-producer machine should never need `-w`, `--producer-key`, `--data-dir`, or `sudo` for daily operations.
2. **One canonical base directory** per network, owned by a dedicated group. Both the interactive user and the service user are members.
3. **Privilege separation.** Private keys are readable only by the group. Chain data is writable by the service. Binaries are owned by root (installed once).
4. **Explicit overrides** for multi-node or nonstandard setups, but the default path is always tried first.

## 1. Standard Directory Layout

### Base path convention

```
/var/lib/doli/{network}/         # Linux (FHS-compliant)
~/Library/Application Support/doli/{network}/   # macOS (user-scoped)
```

The CLI and node resolve the base path with this logic (in order):

1. `--data-dir` flag (explicit override -- wins always)
2. `DOLI_DATA_DIR` environment variable
3. Platform default:
   - **Linux**: `/var/lib/doli/{network}`
   - **macOS**: `~/Library/Application Support/doli/{network}`

For backward compatibility, if `/var/lib/doli/{network}` does not exist but `~/.doli/{network}` does, use the legacy path and print a one-time migration hint.

### Directory tree

```
/var/lib/doli/mainnet/           # Base for mainnet
  wallet.json                    # Producer wallet (keypair + BLS)
  node_key                       # libp2p identity (auto-generated)
  config.toml                    # Optional: node config overrides
  state_db/                      # RocksDB chain state
  block_store/                   # Block headers + bodies
  undo/                          # Rollback undo data
  archive/                       # Block archive (optional, seed only)

/var/lib/doli/testnet/           # Same structure for testnet
/var/lib/doli/devnet/            # Same structure for devnet

/usr/local/bin/doli              # CLI binary
/usr/local/bin/doli-node         # Node binary

/var/log/doli/mainnet.log        # Node log (systemd can also use journal)
/var/log/doli/testnet.log
```

### Layout table

| Item | Path (Linux) | Owner | Mode | Notes |
|------|-------------|-------|------|-------|
| CLI binary | `/usr/local/bin/doli` | root:root | 0755 | Installed by `doli install` or package |
| Node binary | `/usr/local/bin/doli-node` | root:root | 0755 | Same |
| Base dir | `/var/lib/doli/{net}/` | doli:doli | 0750 | Created by `doli init` or installer |
| Wallet | `/var/lib/doli/{net}/wallet.json` | doli:doli | 0640 | Private keys inside -- group-readable |
| Node key | `/var/lib/doli/{net}/node_key` | doli:doli | 0640 | libp2p identity |
| State DB | `/var/lib/doli/{net}/state_db/` | doli:doli | 0750 | RocksDB |
| Block store | `/var/lib/doli/{net}/block_store/` | doli:doli | 0750 | On-disk blocks |
| Config | `/var/lib/doli/{net}/config.toml` | doli:doli | 0644 | Optional overrides |
| Log dir | `/var/log/doli/` | doli:doli | 0750 | Service logs |
| Service file | `/etc/systemd/system/doli-{net}.service` | root:root | 0644 | Generated by `doli service install` |

### macOS layout

| Item | Path (macOS) | Notes |
|------|-------------|-------|
| Base dir | `~/Library/Application Support/doli/{net}/` | User-scoped, no system user needed |
| Wallet | `{base}/wallet.json` | 0600 (user-only, no group model) |
| Log | `{base}/node.log` | Written by launchd agent |
| Plist | `~/Library/LaunchAgents/network.doli.{net}.plist` | User agent, not system daemon |

## 2. User/Group Model

### Linux

| Entity | Type | Purpose |
|--------|------|---------|
| `doli` | System user | Runs the doli-node service. No login shell. Home = `/var/lib/doli` |
| `doli` | System group | Grants access to wallet + data. The interactive user is added to this group |

**Setup** (done once by installer):

```bash
# Create system user + group
sudo useradd --system --home-dir /var/lib/doli --shell /usr/sbin/nologin --create-home doli

# Add the interactive user to the doli group
sudo usermod -aG doli $USER

# Create network directories
sudo install -d -o doli -g doli -m 0750 /var/lib/doli/mainnet
sudo install -d -o doli -g doli -m 0750 /var/log/doli
```

After adding the user to the group, the user must log out and back in (or `newgrp doli`) for the group to take effect.

**Why a dedicated user?**
- The service does not run as root. No root escalation vector.
- The service does not run as the interactive user. A compromised node binary cannot access the user's SSH keys, GPG keys, etc.
- The interactive user (member of `doli` group) can read the wallet for signing transactions.
- Private keys are 0640 (owner=doli, group=doli). Only the service and group members can read them.

### macOS

No system user needed. The node runs as the current user via a launchd user agent. Wallet permissions are 0600 (user-only). macOS does not have the multi-user security concern of a shared Linux server.

## 3. Default Path Resolution

### Wallet discovery (CLI: `doli send`, `doli producer register`, etc.)

The `doli` CLI resolves the wallet path as follows:

1. `-w / --wallet` flag (explicit)
2. `DOLI_WALLET_FILE` env var
3. `{base_dir}/wallet.json` where `base_dir` is resolved per Section 1

This means `doli send doli1abc 10` just works -- no `-w` needed.

### Producer key discovery (Node: `doli-node run --producer`)

The `doli-node` resolves the producer key as follows:

1. `--producer-key` flag (explicit)
2. `DOLI_PRODUCER_KEY` env var
3. `{data_dir}/wallet.json` where `data_dir` is the node's data directory

When `--producer` is set and no key is found at any location, the node exits with a clear error:

```
ERROR: --producer requires a wallet with producer keys.
       Expected at: /var/lib/doli/mainnet/wallet.json
       Create one:  doli new
       Or specify:  --producer-key /path/to/wallet.json
```

### Data directory discovery (Node + CLI)

Both binaries resolve data_dir identically (Section 1, repeated for clarity):

1. `--data-dir` flag
2. `DOLI_DATA_DIR` env var
3. Platform default for the given `--network`

### RPC endpoint discovery (CLI)

1. `-r / --rpc` flag
2. `DOLI_RPC_URL` env var
3. Network default (`http://127.0.0.1:8500` for mainnet, `18500` for testnet, `28500` for devnet)

No change from current behavior -- already works well.

## 4. Sudo Model

### Goal: zero sudo for daily operations

| Operation | Needs sudo? | Why |
|-----------|------------|-----|
| `doli new` | No | Writes to `/var/lib/doli/{net}/wallet.json` (user is in `doli` group) |
| `doli balance` | No | Read-only RPC call |
| `doli send` | No | Reads wallet (group-readable), sends via RPC |
| `doli producer register` | No | Same as send |
| `doli producer status` | No | Read-only RPC call |
| `doli chain` | No | Read-only RPC call |
| `doli snap` | No | Writes to data_dir (group-writable), restarts service via `systemctl --user` or polkit |
| `doli upgrade` | **Yes** | Writes to `/usr/local/bin/` (root-owned) |
| `doli service install` | **Yes** | Writes to `/etc/systemd/system/` |

### Eliminating sudo from `doli snap`

Currently `doli snap` calls `sudo systemctl stop/start`. With the new model:

**Option A (recommended): polkit rule.** Install a polkit rule that allows members of the `doli` group to start/stop `doli-*` services without a password.

```
/etc/polkit-1/rules.d/50-doli.rules:
polkit.addRule(function(action, subject) {
    if (action.id == "org.freedesktop.systemd1.manage-units" &&
        action.lookup("unit").indexOf("doli-") == 0 &&
        subject.isInGroup("doli")) {
        return polkit.Result.YES;
    }
});
```

Then `doli snap` calls `systemctl stop doli-mainnet` (no sudo).

**Option B: sudoers entry.** A more traditional approach:

```
/etc/sudoers.d/doli:
%doli ALL=(ALL) NOPASSWD: /usr/bin/systemctl stop doli-*, /usr/bin/systemctl start doli-*, /usr/bin/systemctl restart doli-*
```

Both options are installed by `sudo doli service install`.

### Eliminating sudo from `doli upgrade`

This is the one operation that genuinely needs elevated privileges (writing to `/usr/local/bin/`). Two approaches:

**Approach 1: Keep `sudo doli upgrade`.** Simple, honest, understandable. The producer runs it rarely (only on new releases). One command.

**Approach 2: Self-elevating binary.** `doli upgrade` detects it lacks write permission, then re-execs itself with `sudo`. The user sees a password prompt. This is what `brew` does.

**Recommendation**: Approach 1. Producers upgrade rarely. `sudo doli upgrade --yes` is clear. Trying to hide sudo adds complexity and confuses debugging.

### Summary: what the producer actually types

Daily operations (no sudo):
```bash
doli balance
doli send doli1abc 100
doli producer status
doli rewards list
doli rewards claim-all
```

Rare operations (sudo):
```bash
sudo doli upgrade --yes
sudo doli service install
```

## 5. Install Flow (New Producer, From Zero)

### Step 1: Install binaries

```bash
curl -fsSL https://get.doli.network | bash
```

The install script:
1. Downloads the latest release tarball
2. Verifies SHA256 checksum
3. Verifies maintainer signatures (3/5)
4. Installs `doli` and `doli-node` to `/usr/local/bin/`
5. Creates the `doli` system user and group (Linux only)
6. Adds the current user to the `doli` group
7. Creates `/var/lib/doli/mainnet/` with correct ownership
8. Creates `/var/log/doli/` with correct ownership
9. Prints: "Log out and back in for group membership to take effect, then run: doli init"

### Step 2: Initialize (create wallet + service)

```bash
doli init
```

This single command:
1. Checks that the user is in the `doli` group (prompts to re-login if not)
2. Creates a new wallet at `/var/lib/doli/mainnet/wallet.json`
3. Displays the 24-word seed phrase with a bold warning to write it down
4. Waits for user to confirm they saved the phrase
5. Generates the BLS attestation key (auto, no separate `add-bls` step)
6. Prints the producer's public key and address
7. Prints: "Your wallet is ready. Next: fund your address with 10+ DOLI, then run: doli producer register"

If the wallet already exists, `doli init` prints:
```
Wallet already exists at /var/lib/doli/mainnet/wallet.json
  Address: doli1abc...
  Run 'doli info' to see details.
  Run 'doli init --force' to overwrite (DANGEROUS: destroys existing keys).
```

### Step 3: Fund the wallet

The producer receives DOLI from another party (exchange, OTC, faucet for testnet):
```bash
doli balance    # Check if funds arrived
```

### Step 4: Register as producer

```bash
doli producer register --bonds 1
```

This:
1. Finds wallet at the default path
2. Builds a Register transaction (1 bond = 10 DOLI)
3. Signs and submits via RPC
4. Prints: "Registration submitted. Active after 10 blocks (~100 seconds)."

### Step 5: Install and start the service

```bash
sudo doli service install
```

This:
1. Generates the correct systemd unit file (or launchd plist)
2. Uses the standard paths -- no user-specific content
3. Installs the polkit rule (or sudoers entry) for service control
4. Enables the service
5. Starts the service
6. Prints: "doli-node is running. Check status: doli producer status"

### Step 6: Verify

```bash
doli producer status
doli chain
```

### Complete flow (6 commands):

```bash
curl -fsSL https://get.doli.network | bash   # 1. Install
# (log out, log back in)
doli init                                      # 2. Create wallet + keys
doli balance                                   # 3. Check funds
doli producer register --bonds 1               # 4. Register
sudo doli service install                      # 5. Start node
doli producer status                           # 6. Verify
```

## 6. Service File Templates

### Linux (systemd)

Generated by `doli service install`. No user-editable paths.

```ini
[Unit]
Description=DOLI Mainnet Node
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=doli
Group=doli
ExecStart=/usr/local/bin/doli-node \
  --network mainnet \
  run \
  --producer \
  --yes --force-start
Restart=always
RestartSec=10
StartLimitIntervalSec=600
StartLimitBurst=5

# Logging
StandardOutput=append:/var/log/doli/mainnet.log
StandardError=append:/var/log/doli/mainnet.log

# Security hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/lib/doli /var/log/doli
PrivateTmp=true
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
```

Key observations:
- **No `--data-dir`**: The node resolves it from the platform default (`/var/lib/doli/mainnet/`)
- **No `--producer-key`**: The node finds `wallet.json` in the data directory
- **No `--rpc-port`**: Uses the network default (8500)
- **No `--p2p-port`**: Uses the network default (30300)
- **`User=doli`**: Runs as the dedicated system user
- **`ProtectHome=true`**: Cannot access any user's home directory
- **`ProtectSystem=strict` + `ReadWritePaths`**: Can only write to its own directories

### macOS (launchd)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>network.doli.mainnet</string>
    <key>ProgramArguments</key>
    <array>
        <string>/usr/local/bin/doli-node</string>
        <string>--network</string>
        <string>mainnet</string>
        <string>run</string>
        <string>--producer</string>
        <string>--yes</string>
        <string>--force-start</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
    <key>StandardOutPath</key>
    <string>/Users/USERNAME/Library/Application Support/doli/mainnet/node.log</string>
    <key>StandardErrorPath</key>
    <string>/Users/USERNAME/Library/Application Support/doli/mainnet/node.log</string>
    <key>SoftResourceLimits</key>
    <dict>
        <key>NumberOfFiles</key>
        <integer>65535</integer>
    </dict>
</dict>
</plist>
```

`USERNAME` is substituted at install time by `doli service install`.

## 7. Multi-Node Support (Advanced)

For operators running multiple producers on a single machine (the current internal setup), every default can be overridden.

### Environment variable overrides

| Variable | Effect |
|----------|--------|
| `DOLI_DATA_DIR` | Override base data directory |
| `DOLI_WALLET_FILE` | Override wallet path |
| `DOLI_PRODUCER_KEY` | Override producer key path |
| `DOLI_RPC_URL` | Override RPC endpoint for CLI |
| `DOLI_NETWORK` | Override network (mainnet/testnet/devnet) |

### CLI flag overrides

| Binary | Flag | Purpose |
|--------|------|---------|
| `doli` | `-w /path/to/wallet.json` | Use a specific wallet |
| `doli` | `-r http://host:port` | Connect to a specific node's RPC |
| `doli` | `-n testnet` | Use testnet network |
| `doli-node` | `--data-dir /path/` | Use a specific data directory |
| `doli-node` | `--producer-key /path/` | Use a specific producer key |
| `doli-node` | `--p2p-port 30301` | Override P2P port |
| `doli-node` | `--rpc-port 8501` | Override RPC port |

### Multi-node service example

For the internal deployment (N1-N12), the `install-services.sh` script continues to work as-is. The only change is that single-producer external operators never need to touch it.

```bash
# Internal operator creating N2 service manually:
sudo doli service install \
  --name doli-mainnet-n2 \
  --data-dir /mainnet/n2/data \
  --producer-key /mainnet/n2/keys/producer.json \
  --p2p-port 30302 \
  --rpc-port 8502
```

### Multi-node CLI usage

```bash
# Talk to N2's RPC
doli -r http://127.0.0.1:8502 -w /mainnet/n2/keys/producer.json balance

# Or set env vars in a shell profile/alias
alias doli-n2='doli -r http://127.0.0.1:8502 -w /mainnet/n2/keys/producer.json'
doli-n2 balance
```

## 8. `doli init` Detailed Behavior

`doli init` is a new command that replaces the manual `doli new` + `doli add-bls` + service setup dance.

```
doli init [--network <mainnet|testnet|devnet>] [--data-dir <path>] [--non-producer]
```

### Flow

```
$ doli init

  DOLI Producer Setup
  ====================

  Network:  mainnet
  Data dir: /var/lib/doli/mainnet/

  Creating wallet...

  YOUR 24-WORD SEED PHRASE:
  ┌──────────────────────────────────────────────────────┐
  │  1. abandon  2. ability  3. able  ... 24. zoo       │
  │                                                      │
  │  WRITE THIS DOWN AND STORE IT SAFELY.                │
  │  This is the ONLY way to recover your wallet.        │
  │  It will NOT be shown again.                         │
  └──────────────────────────────────────────────────────┘

  Have you saved your seed phrase? [y/N]: y

  Generating BLS attestation key...

  Your producer identity:
    Address:    doli1a2b3c4d...
    Public key: 04abc123...
    BLS key:    8a7f3e...

  Wallet saved to: /var/lib/doli/mainnet/wallet.json

  NEXT STEPS:
  1. Fund your address with at least 10 DOLI (1 bond)
  2. Register: doli producer register --bonds 1
  3. Start:    sudo doli service install
```

### What `doli init` does NOT do

- Does not start the node (that is `doli service install`)
- Does not register as a producer (that is `doli producer register`)
- Does not require sudo (writes to group-writable directory)
- Does not overwrite an existing wallet (unless `--force`)

## 9. `doli service` Subcommand

New subcommand group for service lifecycle:

```
doli service install   [--network] [--name] [--data-dir] [--producer-key] [--p2p-port] [--rpc-port]
doli service uninstall [--network] [--name]
doli service start     [--name]
doli service stop      [--name]
doli service restart   [--name]
doli service status    [--name]
doli service logs      [--name] [--follow] [--lines N]
```

Default `--name` is `doli-{network}` (e.g., `doli-mainnet`).

`install` requires sudo. All others work without sudo (via polkit/sudoers).

## 10. Migration Path

Existing installations use `~/.doli/{network}/` and have no `doli` system user.

### Backward compatibility

1. If `/var/lib/doli/{net}/` does not exist but `~/.doli/{net}/` does, use the legacy path.
2. Print a one-time warning: "Using legacy data directory ~/.doli/mainnet/. Run 'doli migrate' to move to the standard location."
3. `doli migrate` (new command) moves data and wallet to `/var/lib/doli/{net}/`, fixes ownership, and updates the service file.

### Internal deployment

The internal multi-node deployment (`/mainnet/n1/`, `/mainnet/n2/`, etc.) is unaffected. Those nodes use explicit `--data-dir` and `--producer-key` flags in their service files. The new defaults only apply when no flags are provided.

## 12. Security Considerations

### Private key protection

| Threat | Mitigation |
|--------|-----------|
| World-readable wallet | `wallet.json` is 0640 (Linux) or 0600 (macOS) |
| Service reads user's home dir | `ProtectHome=true` in systemd unit |
| Compromised node binary accesses SSH keys | Node runs as `doli` user, not interactive user |
| Root escalation via node | `NoNewPrivileges=true` in systemd unit |
| Wallet visible in process args | Wallet path is resolved internally, never appears in `ps` output |
| Backup seed phrase exposure | Seed phrase is shown once at `doli init`, never stored on disk |

### Service hardening (Linux systemd)

```ini
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/lib/doli /var/log/doli
PrivateTmp=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectControlGroups=true
RestrictRealtime=true
RestrictSUIDSGID=true
```

### Wallet encryption (future enhancement)

Currently the wallet stores private keys in plaintext JSON. A future enhancement would add password-based encryption (argon2id + AES-256-GCM). This is out of scope for this proposal but the directory layout supports it -- the encrypted wallet replaces the plaintext one at the same path.

## 13. Implementation Plan

Modify the path resolution in both `doli` CLI and `doli-node` to use the new discovery chain (flag > env > platform default > legacy fallback). No breaking changes -- existing `~/.doli/` installations continue to work.

**Files to modify:**

Add the unified initialization command that combines `new` + `add-bls`.

**Files to add/modify:**

## 14. Design Decisions

| Decision | Alternatives Considered | Justification |
|----------|------------------------|---------------|
| Dedicated `doli` system user | Run as interactive user; Run as root | Principle of least privilege. Root is too powerful. Interactive user leaks access to SSH keys, etc. |
| `/var/lib/doli/` base path | `~/.doli/`; `/opt/doli/`; `/etc/doli/` | FHS-compliant. `/var/lib/` is for variable application data. `~` has the tilde-resolution and root-vs-user problem. |
| Group-based wallet access | SUID binary; sudo for all wallet ops; Socket-based wallet daemon | Group membership is simple, auditable, and requires no running daemon. SUID is a security risk. |
| polkit for service control | sudo-only; SUID wrapper; socket activation | polkit is the modern Linux way to grant unprivileged users specific systemd actions. sudoers is the fallback for older systems. |
| `doli init` combines new + add-bls | Separate commands | The producer should never have to know about BLS as a separate step. One command, one seed phrase, one identity. |
| Wallet in data dir (not separate) | `~/.doli/wallet.json` global; `/etc/doli/keys/` | Co-locating wallet with data means `--data-dir` is the only override needed. Simpler mental model. |
| Keep `sudo doli upgrade` | Self-elevating; User-writable binary dir | Honest about what needs privilege. Producers upgrade rarely. Self-elevation is magic that breaks in containers. |

## 15. Compatibility Matrix

| Scenario | Before (current) | After (this proposal) |
|----------|------------------|----------------------|
| Single producer, Linux | `doli -w ~/.doli/wallet.json send ...` | `doli send ...` |
| Single producer, macOS | Same, but `~/` varies with service user | `doli send ...` |
| Multi-node operator | Explicit `-w` and `--data-dir` everywhere | Same (flags still work) |
| Service file | User-specific absolute paths | Generic, no user paths |
| Wallet created by root, used by user | Broken (wrong `~`) | Works (shared group) |
| `doli snap` | `sudo doli snap` | `doli snap` |
| `doli upgrade` | `sudo doli upgrade` | `sudo doli upgrade` (unchanged, honest) |
| Existing `~/.doli/` installations | Works | Works (legacy fallback + migration path) |
