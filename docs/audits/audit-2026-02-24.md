## Executive Summary

**Status: REQUIRES CHANGES**

14 critical findings, 12 high-severity findings, ~30 minor findings, and significant specs/docs drift across the codebase.

The most severe cluster is in the **consensus validation pipeline**: `validate_block()` is never called in production, coinbase amounts are never validated, and pubkey_hash comparison uses only 20 of 32 bytes. Together, these mean a malicious producer can forge arbitrary blocks that all nodes accept without question.

The second cluster is in **crash safety**: file-based persistence (chain_state.bin, producers.bin) is not atomic, and RocksDB write paths use `expect()` that crash the node on disk errors.

The third cluster is in **VDF correctness**: Wesolowski proof generation is broken for production-scale iterations (t > 128), blocking new producer registration on mainnet.

## Priority Triage

### P0 ŌĆö Fix immediately (consensus/security critical)

1. **CORE-C1 + CORE-C2**: Wire `validate_block()` into production block processing. This is the single most important fix.
2. **STOR-C1**: Atomic file persistence (write-tmp + fsync + rename pattern)
3. **RPC-C1**: Add signature verification to `submitVote`
4. **NET-C1**: Consume rate limiter tokens on malformed messages

### P1 ŌĆö Fix before next deployment

5. **CRYPTO-C1**: Fix `div_2pow_by_l` (q=0 not q=1) ŌĆö blocks new producer registration
6. **CRYPTO-C2 + CORE-C3**: Unify pubkey_hash to one function, compare all 32 bytes
7. **STOR-C2**: Replace `expect()` on RocksDB writes with error propagation
8. **NET-C2**: Reduce MAX_SYNC_SIZE from 512MB to ~50MB
9. **RPC-H5**: Guard against `DOLI_TEST_KEYS=1` on mainnet

## Final Verdict

**The chain operates today because all 5 producers are trusted (bootstrap maintainers).** The moment an untrusted producer joins or a producer is compromised, the lack of block validation (CORE-C1/C2) becomes immediately exploitable. This is the highest-priority fix.

The VDF bug (CRYPTO-C1) is a blocker for network growth ŌĆö no new producers can register on mainnet until it's fixed.

The crash-safety issues (STOR-C1/C2) are a ticking time bomb ŌĆö any power failure during state write corrupts the node.
